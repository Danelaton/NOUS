# ADR-003: Document Knowledge System

## Status
Accepted

## Date
2026-04-14

## Context
NOUS agents need to read and maintain both formal architectural decisions (ADRs) and informal technical documentation (logs, references). The system must make documents discoverable and the agent accountable for maintaining the index.

## Decisions

### 1. Two Documentation Layers

**Decision**: Documentation is split into two distinct layers with different ownership:

**Layer 1 — Formal (tracked in git):**
- Location: `docs/`
- Format: ADRs (Architecture Decision Records) named `ADR_XXX_topic.md`
- Owner: Team (tracked, reviewed, approved)
- Purpose: Record architectural decisions with full rationale

**Layer 2 — Informal (NOT tracked in git):**
- Location: `dev/docs/`
- Format: Free-form logs, references, team context
- Owner: Agent (maintained during sessions)
- Purpose: Technical context, migrations, troubleshooting

**Rationale**:
- Formal docs belong in git — they represent team agreements
- Informal docs are session artifacts — no value in tracking
- Clear ownership prevents confusion

### 2. Document Reading Triggers

**Decision**: The agent reads specific documents based on the situation, not indiscriminately.

| Situation | Document to Read | Why |
|-----------|------------------|-----|
| Before modifying existing code | ADR of the topic + `dev/docs/troubleshooting.md` | Past decisions + known issues |
| Before architectural decision | All relevant ADRs in `docs/` | Don't repeat decisions |
| New concept/system | Full `dev/docs/` | Technical context of codebase |
| User mentions team/person | `dev/docs/team_context.md` | Social and technical context |
| Work touching DB | `dev/docs/migration_log.md` | Migration history |
| Blocked on issue | `dev/docs/troubleshooting.md` + relevant ADRs | Prior solutions |
| New project for agent | Complete `docs/` | Load all historical decisions |

### 3. docs_index.md — Auto-Generated Map

**Decision**: `.agent/docs_index.md` is auto-generated by the agent and maps all documentation.

**Agent maintains it**:
- New ADR created → add entry
- New file in `dev/docs/` → add entry
- `dev/docs/` file updated → update "last updated" column
- ADR count in index matches actual `docs/` count

**Rationale**:
- Single source of document location
- Agent never searches blindly
- Index update is part of session close protocol

### 4. Post-Session Maintenance Protocol

**Decision**: At the end of every session, the agent:

1. REVIEW `dev/docs/` — update any files that changed this session
2. UPDATE docs_index.md — add/remove entries if files were added/removed
3. VERIFY ADR count in docs_index matches actual `docs/` count
4. LOG session summary in `dev/docs/session_log.md` (append-only)

### 5. Forbidden Rules

**Decision**: The following are explicitly forbidden:

- Modifying a tracked ADR after creation — create a new ADR instead
- Deleting `dev/docs/` files without logging content to Session Log first
- Skipping relevant ADR review before making architectural suggestions

**Rationale**:
- ADRs are immutable — changes create new records
- Deletion loses institutional knowledge
- Agent must always load relevant context before advising

## Consequences

### Positive
- Clear distinction between formal and informal documentation
- Agent maintains index without user intervention
- Triggers ensure relevant documents are always read
- Immutable ADRs create full audit trail

### Negative
- Agent must self-govern document reading discipline
- Two documentation locations require maintenance awareness
- docs_index can drift from reality if agent skips updates
